# IVD Shiny 应用性能优化 — 现状总结（给自动化 Agent）

本文档说明当前代码里的**性能时间戳脚手架**、已采取的优化思路、**仍可疑的耗时区间**与**协作时需注意的约束**，便于后续自动迭代性能工作。

---

## 1. 项目与运行方式（背景）

- **技术栈**：R Shiny 单文件主应用 `app.R`（约 6200+ 行）+ PostgreSQL + `pool` 连接池 + Nginx（工号经 `X-User` 头传入）。
- **部署**：多用户共享**同一 R 进程**（多会话），4 个 `app-shiny` 容器负载均衡；调试问题需**看全 4 个容器日志**（见 `CLAUDE.md`）。
- **辅助代码**：`R/ivd_server_helpers.R` 在 `app.R` 里 **`source(..., local = environment())`** 加载一次（与 `APP_TZ_CN`、`norm_progress` 等处在**同一 Shiny 应用环境**内，避免 `source()` 默认落到 `.GlobalEnv` 导致闭包找不到变量）。
- **Docker**：`Dockerfile` / `Dockerfile.update` 需 **`COPY R/ /app/R/`**，否则容器内 `source("R/ivd_server_helpers.R")` 失败。

## 2. 性能脚手架：`ivd_perf_ts` / `ivd_perf_elapsed`

定义位置：`app.R` 约第 19–28 行。

| 函数 | 行为 | 用途 |
|------|------|------|
| `ivd_perf_ts(label)` | 打印一行带**墙钟时间**的日志，前缀固定为 `[IVD_PERF]` | 标记事件发生的绝对时刻；或配合 `on.exit` 打印**某段代码块自身**的 `elapsed_s`（见下） |
| `ivd_perf_elapsed(t0, label)` | 打印墙钟时间 + **`(+s=…)`**，其中秒数为 **`(proc.time() - t0)[3]`**（CPU/Elapsed 混合口径，R 默认） | 相对某一 `t0` 的累计耗时，常用于相对 **`server()` 进入时刻** 或相对**局部 `t0`** |

**日志检索**：在容器日志中搜索 **`IVD_PERF`** 即可串联时间线。

**注意**：

- `+s=` 的参照点取决于传入的 `t0`。会话级主线使用 **`t_ivd_sess0 <- proc.time()`**（在 `server()` 入口、`server() enter` 日志之后立即设置）。
- 部分子段落使用**局部** `t0`（如 `t0_fs`、`t0_tv`），其 `+s=` 表示相对该局部起点，**不要与 `t_ivd_sess0` 混淆**。

## 3. 关键时间点含义（建议 Agent 按此解读日志）

### 3.1 应用加载（全局，每个 R 进程一次）

- `GLOBAL after dbPool()` — 连接池就绪  
- `GLOBAL after ui fluidPage() built` — UI 对象构建完成  
- `GLOBAL after source(ivd_server_helpers.R)` — 辅助函数已进入应用环境  
- `GLOBAL before shinyApp() return` — `app.R` 顶层即将结束、即将进入 `shinyApp()`

**与首屏墙钟的关系**：用户感知的「第一次打开很慢」往往还包含 **R 冷启动、加载包、执行上述全局段**；这些**不一定**出现在 `server() enter` 之后的 `+s=` 里。

### 3.2 每个浏览器会话：`server()` 内

- **`server() enter (new session)`** — 新会话开始执行 `server()`。  
- **`t_ivd_sess0`** — 紧接着设置，作为本会话多条 `ivd_perf_elapsed(t_ivd_sess0, …)` 的基准。

- **`sess|onFlushed#1 (first Shiny cycle to client) (+s=…)`**  
  - 通过 `session$onFlushed(..., once = TRUE)` 注册。  
  - 含义：**从进入 `server()` 起，到第一次「整轮 Shiny reactive 输出刷到客户端」完成**。  
  - 这是衡量「首屏可交互前 R 侧有多忙」的核心外显指标之一，**包含**：`server()` 整段注册 + 首轮 reactive/observer 执行 + 序列化/推送等，不单指甘特图。

- **`sess|after reactiveVal / session init block` / `sess|after in-server function defs (before first output$)`**  
  - 历史遗留命名：大段逻辑已迁出 `server()` 后，这两行之间几乎只剩 `reactiveVal` 等，**耗时应接近 0**；若仍看到较大 `+s=`，说明中间被插入了重逻辑，需核对是否改动了顺序。

- **`sess|server() body finished registering (about to return from server fn) (+s=…)`**  
  - 打在 `server()` **函数体末尾**（所有 `output$` / `observe*` / `reactive({})` 注册完成之后）。  
  - **`+s=` ≈ 整段 `server()` 同步执行耗时**（大量闭包构建 + 注册），这是已观测到的**主要瓶颈之一**（例如约 **18s** 量级，视机器与代码量而定）。

### 3.3 首轮业务输出（带 `sess|` 且常用 `.ivd_sess_once` 防重复打点）

下列日志仅在**第一次**命中对应分支时打印（通过 `server` 内 `.ivd_sess_once` 环境记录 flag）：

| 日志标签 | 含义 |
|----------|------|
| `sess|output$gantt_db_msg renderUI first` | 首次执行该 `renderUI` |
| `sess|output$app_title_panel renderUI first` | 同上 |
| `sess|output$current_user_display renderUI first` | 同上 |
| `sess|current_user_auth FIRST run` | `reactive current_user_auth` 首次执行（查 `05人员表` 等） |
| `sess|gantt_filter_options reactive FIRST run` | 甘特筛选下拉选项 reactive 首次跑（多路 DISTINCT SQL） |
| `sess|observe(gantt_filter updateSelect) FIRST fire` | 更新筛选 `selectInput` 的 `observe` 首次触发 |
| `sess|output$my_gantt renderTimevis FIRST` | 甘特 `renderTimevis` 首次执行 |
| `sess|output$meeting_tab_ui renderUI first` | **仅当用户切换到「会议决策」Tab** 后首次构建会议 UI（已做延迟，不应在纯甘特首屏出现） |

### 3.4 甘特数据管线（无 `sess|` 前缀居多，部分带 `elapsed_s`）

典型顺序（简化）：

- `gantt_sql_filter_bundle BEGIN/END` — 拼筛选与权限相关 WHERE/参数  
- `stage_definition_df BEGIN/END` — 读 `08` 阶段定义  
- `gantt_data_db BEGIN/END` — 视图 `v_项目阶段甘特视图` 等  
- `gantt_data_all_stages BEGIN/END` — 全阶段视图  
- `processed_data BEGIN/END` — 组装 timevis 的 groups/items（内部有多条细分日志）  
- `renderTimevis(my_gantt) BEGIN` → `after processed_data()` → `timevis() widget build` → `after timevis()`

**已验证结论**：上述甘特+timevis 路径往往在 **1s 内**，**不是** `onFlushed` 总时长的主要构成；主要矛盾在 **`server()` 注册** 与 **renderTimevis 结束到 onFlushed 之间的未打点区间**。

## 4. 当前性能优化思路（已做 + 方向）

### 4.1 已做

1. **把不依赖 `input` / `output` / `session` 的大块纯函数**迁到 `R/ivd_server_helpers.R`，在应用加载时 `source` **一次**，减少**每个会话**在 `server()` 里重复构造巨型闭包。  
2. **`source(..., local = environment())`**：保证与 `APP_TZ_CN` 等变量同环境，避免 `today_beijing()` 等报「找不到对象」。  
3. **全局 `%||%`**：供顶层与 helpers 共用。  
4. **会议 Tab 延迟**：`req(identical(input$main_tabs, "tab_meeting"))`，避免首屏构建会议相关 UI 及 `meeting_filter_options` / `meeting_data` 等。  
5. **`make_entry_key` 等**：显式使用 **`pg_pool`**，禁止在 helpers 里隐式捕获会话级 `pg_con`（多用户安全）。  
6. **备注日期样式**：用 `today_beijing()` 而非捕获会话级 `today`，避免错误与跨日问题。

### 4.2 推荐后续方向（给 Agent）

| 方向 | 说明 |
|------|------|
| **继续削减 `server()` 体积** | 仍将**仅依赖参数 + `pg_pool` + 已全局常量**的 `function(){...}` 移出；**依赖 `reactiveVal` / `input` / `session` / 其他 reactive** 的必须留在 `server()` 或通过参数注入。 |
| **解释「renderTimevis 结束 → onFlushed」空档** | 对**其余** `output$` / 无条件 `observe` 增加与甘特同风格的「FIRST」或分段计时，定位 ~数秒级未知耗时。 |
| **避免跨用户泄漏** | 禁止用全局 `list`/环境按用户 id 做无上限缓存；共享的只有池与只读常量。 |
| **不引入 golem 等新包**（除非团队同意） | 当前约束以**单应用 + 辅助 R 文件**为主。 |

## 5. 已观测现象与「可能有问题的版块」（假设）

以下基于真实日志样本归纳，**随部署环境会变**：

1. **`sess|server() body finished registering` 的 `+s=` 很大（例如 ~18s）**  
   - **版块**：整个 `server()` 函数体 — 大量 `output$… <-`、`observeEvent`、`open_*_modal` 等**会话级闭包**。  
   - **性质**：CPU/解析/分配为主，**不一定是 SQL**。

2. **`renderTimevis … after timevis()` 与 `sess|onFlushed#1` 之间仍有数秒～十余秒空档**  
   - **版块**：首轮仍在执行的其他 **output / observe**、或 **序列化/推送**；当前**未全覆盖打点**。  
   - **建议**：按模块（项目汇总、阶段维护、人员中心、各类 `renderUI`/`DT`）补点。

3. **第一次打开总墙钟 > `onFlushed` 的 `+s=`**  
   - **版块**：进程冷启动、`library()`、全局执行、`dbPool` 首次建连等。

4. **负载均衡**  
   - 多容器下日志分散；性能问题可能只在某一实例出现。

## 6. 多用户与安全约束（性能优化时勿破坏）

- **`pg_con <- pg_pool`**：会话内变量指向**共享池**；查询必须带**权限过滤**（`current_user_auth()`、`allowed_subquery` 等），与业务规则一致。  
- **helpers 中**：需要库连接时优先 **`pg_pool` 或显式 `conn` 参数**，勿假设存在会话级 `pg_con`。  
- **`.ivd_sess_once`**：仅用于**本会话内**一次性诊断打点，不应用于存业务数据。

## 8. 给自动化 Agent 的操作提示

1. 改性能相关代码后：至少 **`Rscript -e 'parse(file="app.R")'`** 与 **`parse(file="R/ivd_server_helpers.R")`**。  
2. 验证 `source` 环境：helpers 依赖 **`norm_progress`**（在 `app.R` 中 **`source(helpers)` 之前**定义）。  
3. 新增 `source` 文件时：同步 **Dockerfile 的 `COPY`**。  
4. 日志对比：固定搜索 **`IVD_PERF`**，区分 **`+s=` 相对 `t_ivd_sess0` 还是相对局部 `t0`**。  
5. 优化后若会议 Tab 行为异常：检查 **`input$main_tabs`** 与 `tabsetPanel(id = "main_tabs")` 的 `value` 是否仍为 `"tab_meeting"`。
